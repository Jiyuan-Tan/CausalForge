This project lays foundations for formalizing causal inference. Rigor and
correctness are the most important thing. Check the formal definitions before
coding.

Autonomous pipelines (e.g. `/causalsmith research`) own their escalation
policies; do not impose interactive-collaboration rules on them.

## Universal rules

- Be concise and accurate in conversation. Don't repeat code; refer to the line
  you change.
- Per-declaration documentation is **docstring-canonical**: the description lives
  in each declaration's Lean docstring, and `doc/API.md` is a derived/orientation
  view. Read the relevant docstrings (or `npm run search`) before specific code;
  use `doc/API.md` for per-submodule orientation.
- `doc/API.md` is large (~8k lines), organized by submodule with `## <n>. <path>`
  headers. Do NOT read it in full: Grep the `^## ` header index, then Read with
  offset/limit. For per-declaration lookup prefer `npm run search` /
  `doc/library_index.json` — the canonical, docstring-derived per-decl view.
- Keep each Lean file focused and normally ≤600 lines; split before ~900 when it
  contains independent topics or proof chains. Each file's overview lives in its
  `/-! -/` module docstring.
- Check your proof with the type checker every time you change the codebase.
  Prefer the `lean-lsp` MCP over `lake build` during iteration
  (`lean_diagnostic_messages`, `lean_goal`, `lean_multi_attempt`,
  `lean_local_search`); fall back to `lake build Causalean.<ModulePath>` for
  import-level rebuilds. Full `lake build` is slow — only when touching many files.
- **Two lean-lsp MCP instances, one per Lake project.** CausalSmith is a separate
  Lake project, so the default server (rooted at the repo root / Causalean) cannot
  resolve CausalSmith imports. Editing under `Causalean/` → `mcp__lean-lsp__*`;
  editing under `CausalSmith/` → `mcp__lean-lsp-causalsmith__*` (see `.mcp.json`).
- **Search the library before writing a new lemma** —
  `cd CausalSmith/tools && npm run search -- "<concept words>"`. Ranked
  declarations with statement, source file, tier flags, and the docstring's NL
  first paragraph. Modes: default concept; `--type "<pattern>"`;
  `--goal "<goal type>"`; `--scope module "<area>"` for orientation. Flags:
  `--cluster panel|exactid|partialid|stat|experimentation|scm`, `--k N`,
  `--semantic`. If the index is stale: `lake build && lake exe library_index`.
- Avoid unnecessary Lean imports. Prefer targeted module imports over umbrellas
  like `import Mathlib`; use `#min_imports` / the `minImports` linter when unsure.
- **Library documentation is docstring-canonical.** Author each declaration's
  plain-English description ONCE, in its Lean docstring (FIRST paragraph = the NL
  translation, written for an econometrician with no Lean background, no Lean
  syntax, self-contained). `doc/API.md`'s per-decl tables live inside
  `<!-- GEN:<module> -->` marker regions regenerated by
  `cd CausalSmith/tools && npm run doc:gen` — never hand-edit inside markers; only
  hand-write cross-cutting narrative and worked examples.
- **Causalean and its library-explorer page move together.** After any change to
  declarations or docstrings: `lake build && lake exe library_index`, then
  `cd CausalSmith/tools && npm run embed:library && npm run lint:embeddings`
  before committing. New files must be reachable from the `Causalean.lean` root
  import graph (`check_library_index` fails on orphans).
- **Headline curation at write time.** A main result (identification theorem,
  estimand characterization, paper-named decomposition, asymptotic
  linearity/normality/rate/optimality result, sharp bound) must be added to
  `headline_theorems` in `doc/library_review/<Area>.json`; technical steps stay
  supporting-tier. Renames/moves must update the sidecar entry.
- **Recording a new module:** (1) write the `.lean` file with a `/-! -/` overview
  and a `/--` docstring on every public declaration; (2) `lake build && lake exe
  library_index`; (3) `cd CausalSmith/tools && npm run embed:library && npm run
  lint:embeddings`; (4) add a `headline_theorems` sidecar entry if it is a main
  result; (5) in `doc/API.md`, hand-add only narrative and drop a
  `<!-- GEN:Causalean.<Module> -->` marker pair, then `npm run doc:gen`.
- When writing math in chat, do NOT use LaTeX (`$...$`, `\cup`, …) — use Unicode
  symbols directly (⊆, ∪, ⊥⊥, ∀, ∃, →, …) and indent displayed equations with 4
  spaces.
- **"Build green" ≠ proved.** `lake` exits 0 WITH sorries, and a targeted
  `lake build <module>` can replay a stale olean over a live error. Before
  claiming a theorem done: FULL build of the module, grep the SOURCE for
  `sorry|admit|native_decide|^\s*axiom`, and run `#print axioms` via
  `lake env lean`.
- When editing pipeline/skill prompts (`*.txt`), be concise and don't overfit:
  state the general rule, not examples tied to the specific instance that
  surfaced the issue.

## Workspace layout (two-package split)

- `Causalean/` (repo root package) — foundational library. Hand-curated, stable.
  Causalean never imports anything from CausalSmith.
- `CausalSmith/` (sibling subdir, own Lake project) — umbrella package holding the
  `CausalSmith research` pipeline. Tooling under `CausalSmith/tools/`, NL specs
  under `CausalSmith/doc/research/`, Lean output under
  `CausalSmith/CausalSmith/{Panel,ExactID,PartialID,Stat,Experimentation,SCM}/`.
  `CausalSmith/doc/USER_MANUAL.md` is the user-facing guide;
  `CausalSmith/doc/SETUP.md` the environment prerequisites.

Build via `lake build` (Causalean) and `lake -d CausalSmith build` (CausalSmith).
Promotion of a CausalSmith lemma to Causalean is a deliberate human step: copy the
statement + proof into Causalean, then have CausalSmith re-import the Causalean
version.

## Module map (orientation index)

### Causalean (foundational)
- `Causalean/Graph/` — DAG, d-separation (Bayes Ball), SWIG, sub-SWIG, c-components
- `Causalean/SCM/` — `CausalModel`, do-calculus rules, Markov properties, factored
  kernels
- `Causalean/PO/` — potential outcome calculus, consistency, counterfactuals;
  ID/Exact/ (backdoor, frontdoor, ATE, DID, LATE, RDD, …) and ID/Partial/
  (Manski family, Balke–Pearl, Lee, Fréchet, …)
- `Causalean/Panel/` — panel substrate: DGPs, assignment, fixed effects,
  regression infrastructure
- `Causalean/Experimentation/` — design-based / randomization inference under
  interference; `DesignBased/` is the shared substrate, each paper a sibling folder
- `Causalean/Stat/`, `Causalean/Estimation/` — semiparametric inference,
  concentration, DML/AIPW, minimax
- `Causalean/ML/` — learning-theoretic foundations (uses vendored FoML)
- `Causalean/Discovery/` — causal discovery (concrete example DAGs live under
  `Causalean/SCM/Examples/`)

### CausalSmith (umbrella package)

CausalSmith research runs split into two phases: **discovery** (propose + solve the
math informally — stages D-1, D0, D0.5) and **formalization** (translate to Lean and
prove — F1, F1.5, F2, F2.5, F3, F3.5, F4, F5). Formalization is the verification
half: an F-stage failure usually means the math claim was wrong, not that the
translator broke.

- `CausalSmith/CausalSmith/{Panel,ExactID,PartialID,Stat,Experimentation,SCM}/` —
  theorem outputs per cluster. The `Stat/` cluster (`stat_*` qids) targets a
  minimax rate / efficiency bound / limit law for a *causal* estimand, where the
  rate/efficiency/limit IS the kernel; a matching minimax converse is one
  eligible frontier advance, not a requirement or default.
- `CausalSmith/tools/` — TS infrastructure: `bin/` (entry points),
  `src/pipeline_stages.ts` + `src/pipeline_support.ts` (dispatcher),
  `src/discovery/` (D-stage), `src/formalization/` (F-stage), `src/shared/`,
  `test/`
- `CausalSmith/doc/research/` — specs, state, and the `_bank/` of completed runs
- `CausalSmith/site/` — library-explorer + papers web app

## Multi-step task management

For tasks with multiple steps, act as a manager in the main thread: decompose the
task into focused chunks and execute them directly unless the user explicitly asks
for subagents. Keep coordination, integration, and hard proof decisions in the
main thread. If a step is too large, break it down further so the main thread can
keep context manageable.
