=== WHAT MAKES A GOOD PROPOSAL ===

- A sharp, THEOREM-LEVEL kernel — a precise claim, not "apply method X to topic Y". It is an identification result (ExactID), a partial-identification/bounds result (PartialID), a panel result (Panel), an estimation/inference result (Stat: a convergence rate, a semiparametric efficiency bound, or a limit law for a causal estimand), or a design-based experimental result (Experimentation: the design is a first-class object and the only randomness is the assignment, with FIXED finite-population potential outcomes — EITHER a limit law + Wald-coverage result for a design-based estimator of a causal effect under a KNOWN randomization, with the limit law DERIVED from primitive design / dependency conditions and never assumed, OR an optimal / efficient design that CHOOSES the randomization to optimize a stated criterion (variance / efficiency / power / regret) for a causal estimand with the optimality CERTIFIED (KKT / matching variance lower bound / saddle), with other design-based shapes admissible too; the estimand is a finite-population causal contrast and identification is given by the design), or a graphical/structural result (SCM: a causal query stated over a DAG / ADMG / selection diagram in the do / σ-calculus language — EITHER a NEW identifying functional `Q = f(P_obs)` that holds under a stated (possibly novel) assumption such as monotonicity / shape, proxies / negative controls, selection nodes, or extra interventional inputs, OR a practical, valid graphical / data-fusion partial-identification bound `Q ∈ [L(P_obs), U(P_obs)]` sound over the compatible-SCM class; completeness (ID lane) and sharpness (pID lane) are preferred but NOT required).
- At least one first-class open-ended question — the genuinely novel content whose truth is still open. A proposal that is ALL routine theorems is almost always self-deception; expose the real open content.
- NOVELTY: name the 1–2 closest published results and state a precise gap a referee would accept as a real advance (not a relabelling, not a routine T=2→general-T extension, not an iid→Markov bump, not a CATE/ATT restatement of an ATE result).
- A concrete CONSUMER for each result: an open problem it unblocks, an applied setting it enables, or a published result it makes a corollary — never "future work" / "broad interest".

The MODE block prepended above this prompt tells you HOW to arrive at the kernel for this version (cold-start ideation, a flag-targeted revise, a pivot to a new seed, a kernel swap, or a construction rebuild). This body defines WHAT to emit and the disciplines every version must satisfy.

=== WHAT TO EMIT (the core JSON) ===

A single JSON object. FORMAL fields:
- `qid`, `specialization`, `cluster` (∈ panel | exactid | partialid | stat | experimentation | scm).
- `symbols`: every named object, typed and defined ONCE before use. `{name, type, space?, sig?, def?, role?, refs?}` (`refs` = other symbols this one's `def` mentions).
- `assumptions`: each ONE pure condition. `{id:"ass:<slug>", kind, condition, free_symbols, constants?, standard|novel, used_by?}`. `free_symbols` is an array of ATOMIC symbol-table keys: every array element must name exactly ONE separately declared `symbols[].name`, with an exact string match. Never put a comma-separated tuple or group of symbols in one element; split every referenced symbol into its own array element.
- `definitions`: classes AND constructions. EVERY definition carries an explicit `construction` (the defining expression, so the notation is never opaque). CLASS: `{id:"def:<slug>", name, construction:"{x : φ(x)}", by_member_properties:[ass ids]}` — `construction` is the set-builder and MUST spell out every index parameter (e.g. the triangular-array `n`); `φ` is a predicate over the member-property assumptions, never a witness. CONSTRUCTED OBJECT: `{id, name, construction:"<formula/algorithm>", inputs?}`.
- `statements`: `{id:"thm|lem|prop|oeq:<slug>", kind:"theorem|lemma|proposition|openendedquestion", statement, depends_on, status:"to-prove", justification, gap, consumer}`. Use `oeq:` for an open-ended question. Legacy `conj:`/`conjecture` validates only for existing cores; do not emit it.
- LATEX CONTRACT: every human-facing string is rendered into a LaTeX document. Author compilable LaTeX rather than pseudo-math: use `\\(...\\)` / `\\[...\\]`, real commands (`\\theta`, `\\pi_{\\exp}`, `\\mathcal M_n`, `\\sum`, `\\le`, `\\in`), no raw underscores in prose, no math-only commands outside math mode, and no literal `\\n` tokens. This applies to symbols, assumptions, definitions, statements, top-level prose, per-statement notes, and bibliography text.
- `target_estimand` — a STRING: the causal target as one short symbolic expression (e.g. `"tau = E[Y(1) - Y(0)]"`), NOT an object.
- `estimand_functional` — a STRING: the identifying formula / bounding functional / regression / estimator+rate as one expression, NOT an object.
- `sampling_model` — an OBJECT: the structured draw, e.g. `{design, units, treatment, overlap, nuisance_estimation}`.
- `bibliography` — an array of `{key, citation}`.
- `comparator_promise_table` — an ARRAY, ONE row per theorem-level comparator the abstract / `related_work` names as a target (published article OR public preprint/manuscript; every source you claim to tighten / equal / beat / depart from). Row: `{comparator_bibkey, comparator_claim (one-line verbatim summary of that sourced result), matched_by ("Theorem N" | "Open-ended Question N" | "unmatched"), match_kind (∈ strict_tightening | iff_frontier | equivalence | non_representability | downgraded_to_informed_by | dropped_from_abstract)}`. The reviewer (SC6 / `N-comparator-drift`) AUDITS this table: every theorem-level comparator the prose calls "extends/sharpens/beats" MUST either be `matched_by` a `statement` phrased as that strict relation, or be `downgraded_to_informed_by`, or `dropped_from_abstract`. A named comparator with no row — or a row asserting a relation no `statement` delivers — FAILS review. An unpublished non-archival abstract/talk/private WIP with no public theorem text is disclosed in prose but does not require a promise row and cannot be a theorem-level target. A published-but-inaccessible result is not ignored; mark its load-bearing comparison externally unverifiable rather than guessing. Fill the structured table; do NOT leave theorem-level comparator positioning implicit in `related_work` prose.

PROSE fields (motivation only — never load-bearing math; rendered verbatim):
- `tldr`: one paragraph summarizing the whole proposal.
- `project_justification`: `{gap, niche, fill}` — the literature GAP, the NICHE it opens, how this project FILLS it.
- `related_work`: coherent prose positioning the work against named published papers (one relevance sentence each).
- per `statement`: `justification` (one line: why this claim / why it matters), `gap` (closest prior art + why this differs), `consumer` (one concrete downstream use).
- `technical_internal_limitation` (optional): an internal diagnostic or counterexample that must remain visible but is explicitly NOT a theorem contribution or claim about a published estimator. It renders as an unnumbered diagnostic section.
- `honest_scope` (optional): what is open vs settled.

=== ASSUMPTION DISCIPLINE (each is ONE pure condition) ===

- ONE relation per assumption. NO "thus / consequently / hence / moreover / likewise", NO "is used by/for" pointers (use `used_by`), NO disclaimers ("no CLT is claimed"), NO omnibus bundling unrelated conditions. A derived consequence is a `lemma`, not part of an assumption.
- EXACTLY ONE of:
  * `standard:{name, cite}` — a named published condition (Hölder, Tsybakov margin, Lindeberg, overlap, monotonicity, completeness, …) whose `cite` resolves to a `bibliography` entry. A recognizable variant counts — name the difference. The bar: a referee recognizes it as a standard modeling condition.
  * `novel:{flag:true, justification}` — a genuinely new condition. The `justification` argues it is REALISTIC and satisfied in real settings (a concrete class/example), stated as the contribution itself — NEVER "it makes the rate/proof work". Do NOT let a target rate dictate assumption strength: a faster rate must come from a better estimator/proof, never a stronger-than-standard hypothesis (a sup-norm where L2 is standard, a known nuisance where estimating it is the point, an oracle clip/trim). If the rate needs such a strengthening, pose the honest object instead (the rate under standard assumptions, slower if need be, or an open-ended question on whether the target is attainable under them).
- PRIMITIVE, NOT PROCEDURAL: a condition that defines class membership must be a property of the LAW and its primitive functionals (the propensity, the outcome regressions, P) — NEVER a property of a NAMED ESTIMATOR. Estimability is a property of the nuisance: write `g_a is L2(P_X)-estimable at rate O_P(r_a)` (existential over estimators), not `‖ĝ_a − g_a‖ = O_P(r_a)` for a specific `ĝ_a`. The estimator is then a theorem/construction INPUT, never a membership condition — folding a named `ĝ` into a class conflates the law with a procedure.
- PRUNE-IMPLIED: an assumption mechanically implied by another, or by a symbol's declared space (e.g. "a Bernoulli mean lies in [0,1]"), is NOT a separate assumption. Drop it.

=== DEFINITION DISCIPLINE (class XOR construction) ===

- A CLASS is carved ONLY `by_member_properties`; its `construction` set-builder must be a membership PREDICATE over those properties (spelling out every index param, e.g. `n`), NEVER the image of a witness/construction.
- A least-favorable WITNESS or an ESTIMATOR is a construction (`construction`, plus `inputs`), NEVER an assumption and NEVER a class.
- Class MEMBERSHIP of a construction is ALWAYS a `statement` with `status:"to-prove"` — never an assumption (assuming "the constructed law belongs to the class" is the classic circular defect).
- NO PROVABILITY COMMENTARY (provability firewall). A `construction` NAMES the object or fixes the quantity — it must NEVER editorialize about whether a downstream target is provable ("left open", "not derivable", "no exponent is fixed by the primitives", "remains open"). The frozen core binds the QUESTION (assumptions, estimand, the statement to prove), never the answer's existence; a def that pre-declares a target open is a self-fulfilling prophecy the solver obeys. A derived quantity (a rate, an exponent) is named here only as "DERIVED by <thm-id>"; its openness, if any, lives in a dedicated open node, never in a def.

=== STATEMENT DISCIPLINE ===

- Author the full claim in core notation. `depends_on` cites only assumptions/definitions/other statements you author — keep it acyclic. Every statement is `status:"to-prove"` with NO `route` and NO proof — you propose, you do not prove.
- An open/construct question is ALWAYS an OPEN-ENDED QUESTION (`kind:"openendedquestion"`, id `oeq:<slug>`) — NEVER a conjecture. Do NOT pre-commit a candidate answer: a committed construction freezes the answer and suppresses the solver, who solves best when free to derive the construction itself. So:
  * Pose the QUESTION as the `oeq:` statement (the property to establish, `status:"to-prove"`), and put a named HANDLE (a technique + the target object, e.g. "design an n^{-1/2} estimator for τ* via bridge functions h∈ℋ defined by …") in the associated `definition.construction` as the route — NOT a candidate expression. The SOLVER derives the actual construction and proves the property.
  * Do NOT fill `definition.construction` with a guessed candidate formula/exponent/family; give the Handle (the starting move) only and let the solver determine the object. A bare "design an estimator for τ*" with no concrete starting move is NOT a publishable question — either give a real Handle or drop it.
  * `kind:"conjecture"` is DEPRECATED — never emit one.

=== PROSE DISCIPLINE ===

- Prose MOTIVATES; it introduces no formal object absent from the formal fields. The claim lives in `statement`/`condition`; the prose explains it.
- Register: top-journal econometrics (Econometrica / JoE / QE / ReStud). Plain declarative sentences, one claim each; one name per object. NOT grant register ("we propose", "this unblocks"), NOT blog register ("interestingly", "remarkably"), NOT survey register.
- Every statement MUST carry non-empty `justification`, `gap`, `consumer`; the core MUST carry `tldr`, `project_justification` (all of gap/niche/fill), and `related_work`.

=== WELL-POSEDNESS SELF-CHECK (run before emitting) ===

- Every symbol named in any statement/assumption is declared in `symbols`.
- NO assumption SUPPLIES the object the kernel characterizes (the bound / threshold / loading / estimand). If plugging an assumption in makes the headline near-immediate, that assumption IS your contribution — make it a `statement` to derive, or derive it from primitives.
- COMPARATOR FIDELITY: for every published comparator you claim to tighten/equal/beat, its defining estimand (weights / influence function / score / formula) must be DERIVABLE from its published spec — do NOT assume the comparator's estimand as one of your assumptions, or you are comparing against a stand-in, not the real result.
- SANITY: at least one reduction the kernel must satisfy under a standard special case (homogeneous effects, T=2, iid, no carry-over) collapses to a known result — state it (in a corollary statement or `honest_scope`).
