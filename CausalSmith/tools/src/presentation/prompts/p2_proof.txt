Rules:
- REVISION BRIEF: if the brief above lists findings (not the "first draft" sentinel), address each one that applies to this proof's PRESENTATION (e.g. clarity, an unexplained step, distracting bookkeeping shown to the reader). Faithfulness to the Lean route stays paramount and overrides any finding: do not drop, reorder, or alter a load-bearing step to satisfy the referee, and keep every `% lean:` comment (those are invisible audit markers, not reader-visible text). Never mention the review or a revision.
- Follow the Lean proof's structure: same case splits, same intermediate bounds, same order of argument. Collapse pure bookkeeping (measurability side goals, coercions, rewriting chains) into a sentence.
- SELF-CONTAINED PROSE: the proof must read as a standard mathematical argument with NO reference to the formalization in reader-visible text. Never write "Lean writes/represents…", "in Lean", "the Lean …", "the formal proof/definition/statement", "machine-checked", or "the formalization". State the mathematics directly — write "the pooled supremum equals the integral …", not "Lean writes the pooled supremum as …". Lean is named ONLY in the invisible `% lean: <decl>` comments, never in prose.
- NO LEAN IDENTIFIERS AS MATH: never put a Lean declaration name into reader-visible math — not a camelCase decl name, not a decl applied to its Lean arguments (e.g. `someSchedule α γ a c n`), not a field projection (e.g. `(someConstant …).field`), not a scaffolding alias. Always use the paper's own notation from the notation table / the statement (a schedule written `q_n`, a rate exponent `r`, a constant `\bar u`, …). A raw Lean identifier renders as un-mathematical text and is the single clearest machine-generated tell; the `% lean:` comment is where the Lean name belongs.
- Each numbered step ends with a LaTeX comment `% lean: <declaration name>` naming the Lean lemma/step it renders.
- Where the proof invokes an auxiliary lemma that has its own environment, cite it by its paper label, not by Lean name.
- When the cited-dependency list is nonempty, invoke that published conclusion with the supplied `\citep{...}` and locator in ordinary mathematical prose. Do not call it an assumption of this paper and do not claim to re-prove it. The theorem-local footnote separately records the machine-verification boundary.
- If the Lean route is unclear at any step, output `UNCLEAR: <what is unclear>` and stop — do not improvise a classical argument.
- Output ONLY the LaTeX proof block: \begin{proof}[Proof of ...] ... \end{proof}.

CROSS-REFERENCES: cite every formal environment only through `\cref{obj:<exact-id>}` (or `\Cref` at sentence start). Cleveref supplies the kind and number, so never write a manual kind word before it and never use legacy `\ref{...}` for any target. Reproduce colon-separated kind prefixes in FULL—e.g. `\cref{obj:prop:oracle-regime-reduction}`—and never emit `\label` yourself or write bare obj_ids in prose or proof titles.
NEVER write \label{obj:...} yourself anywhere — the environment macro defines it automatically; an extra one is a LaTeX duplicate-label error and reads as statement drift.
