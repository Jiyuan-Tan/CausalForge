- Read the Lean proof with your tools before writing; follow its structure: same case splits, same intermediate bounds, same order of argument. Collapse pure bookkeeping (measurability side goals, coercions, rewriting chains) into a sentence.
- SELF-CONTAINED PROSE: each proof must read as a standard mathematical argument with NO reference to the formalization in reader-visible text. Never write "Lean writes/represents…", "in Lean", "the Lean …", "the formal proof/definition/statement", "machine-checked", or "the formalization". State the mathematics directly — write "the pooled supremum equals the integral …", not "Lean writes the pooled supremum as …". Lean is named ONLY in the invisible `% lean: <decl>` comments, never in prose.
- NO LEAN IDENTIFIERS AS MATH: never put a Lean declaration name into reader-visible math — not a camelCase decl name, not a decl applied to its Lean arguments (e.g. `someSchedule α γ a c n`), not a field projection (e.g. `(someConstant …).field`), not a scaffolding alias. Use the paper's own notation from the notation table / the statement; a raw Lean identifier renders as un-mathematical text and is the clearest machine-generated tell (the `% lean:` comment is where the Lean name belongs).
- Lemma proofs are auxiliary: keep each proof proportionate — typically one paragraph to a half page; do not pad.
- Each load-bearing step ends with a LaTeX comment `% lean: <declaration name>` naming the Lean lemma/step it renders.
- Where a proof invokes another result that has its own environment, cite it by its paper label, not by Lean name.
- Use a bibliography citation inside a proof only for a source explicitly listed under that lemma's “Published cited dependencies.” Such a source is an invoked published theorem, not an assumption or contribution of this paper. Otherwise never introduce citations inside proofs.
