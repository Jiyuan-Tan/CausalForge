=== NODE → LEAN MAPPING (one-to-one with the core; one tag per node) ===

Realize each plan node exactly as its `lean_kind` says, and prefix its ONE canonical primary top-level decl with a LINE comment `-- @node: <core id>` (with the colon, matching the graph layer; NOT a `/-- … -/` doc comment — a doc comment would pollute the library index's NL-translation field). When one core node needs several Lean declarations, tag EXACTLY ONE primary declaration and leave every companion/auxiliary declaration untagged. Reusing an `@node` id is a hard graph-refresh error. Tag the env S-block's `variable (...)` block with `-- @env: <S-id>`.

- `assumption` → ONE named `def <lean_name> : Prop := <the core condition>`, tagged `-- @node: ass:…`. A statement that lists this atom in its `hyps` takes it as an inline hypothesis `(h : <lean_name> …)`; a class that lists it in `members` references it as a field type. The SAME `def` serves both — a shared atom is defined once. An `ass:` node is ALWAYS a threaded hypothesis: NEVER also emit a `lemma`/`theorem` whose conclusion IS that Prop (a discharge/reuse-lemma proving the assumption from Causalean substrate), even when the plan's `disposition:"reuse"` points at a Causalean decl that only PARTIALLY fits — discharging a declared assumption over-formalizes the note and leaves F3 a false `needs-substrate` bridge `sorry`. If the plan asks you to discharge an `ass:` node against a decl that does not type-match the atom IN FULL, SYNC-BACK the node to `lean_kind:"assumption"`/`disposition:"define-local"` (the Prop `def`, threaded) and record why.
- `gate:true` statement node → scaffold exactly like an `assumption`: ONE tagged `def <lean_name> : Prop := <the core statement conclusion>`, and consumers listing it in `hyps` take it as an inline hypothesis. It is visible substrate debt, never a `lemma`/`theorem … := by sorry`, never an `axiom`, and never included in the theorem manifest. When `gate_class:"cited"`, author the `def` to **faithfully encode the cited lemma at its `source` `locator`** (so F2.5 can match it against the external source — get the quantifiers/constants/direction right, no vacuity), and put the citation (authors, year, locator, handle) in its docstring. `gate_class:"gated"` (or absent) scaffolds unchanged — its own async proof discharges it later.
- definition, CLASS (`lean_kind:"structure"`, has `members`) → a `structure` (or typeclass) whose fields are the member atoms (each field typed by that member's Prop `def`), tagged `-- @node: def:…`. This is the "complicated, multiple-assumptions" bundle — do NOT scatter it into loose hypotheses, and do NOT emit a separate `a<k>Bundle` per assumption (that rule is retired; bundling happens only at class boundaries the plan names).
- definition, CONSTRUCTION (`lean_kind:"def"`) → a `def` / `noncomputable def` realizing the core `construction`, tagged `-- @node: def:…`.
- statement (`lean_kind:"theorem"|"lemma"`) → a `theorem` (T-block) or `lemma`/`private lemma` (L-block), named `lean_name`, tagged `-- @node: thm:|lem:|…`. Its hypotheses are exactly the plan's `hyps` (each an assumption `def` taken inline, or a class structure taken as a bundle). NEVER declare an L-block `theorem` (F2.5 flags it); `theorem` is reserved for T-blocks. Public-vs-private and lemma-vs-theorem are orthogonal — a shared L-lemma in `Basic.lean` is still `lemma`.
- open-ended residual (`kind:"openendedquestion"` in the core) → a named Prop `def`, tagged `-- @node: oeq:…`, that states the question/residual without proving it. Do NOT emit a theorem for this node, do NOT include it in the theorem manifest, and do NOT make downstream theorem proofs depend on it to close the open question. **This holds REGARDLESS of the plan's `lean_kind` — an `openendedquestion` core node is ALWAYS a Prop `def`, even if the plan says `lean_kind:"theorem"` or D0 marked it `status:"proved"`.** An OEQ rendered as a `theorem … := by sorry` is a permanent unproven assertion of an open question (forbidden); a genuine negative/positive resolution must live in a SEPARATE `thm:`/`prop:` node, leaving the `oeq:` node as the recorded question.

For a `disposition:"reuse"` node/S-block: `import` the named decl and build on it (do NOT re-state it over plain ℝ/Set). For `define-local`: write the decl/structure/lemma yourself (sorry body for lemmas/theorems; full body for defs/structures).

=== IMPLEMENT THE PLAN + SYNC-BACK ===

Implement the plan as written. You MAY adjust when an as-planned decision does not work in Lean — but every adjustment must be written BACK into plan.json so the plan stays one-to-one with the emitted Lean:

- A planned `reuse <decl>` does not actually type-fit where the node uses it → switch that node to `disposition:"define-local"`, `reuse:null`, and Edit plan.json accordingly (record why in `notes`).
- An atom is better realized as a named `def` than inline, or a file split must change → Edit the affected node entry (`target_file`, etc.).
- Keep coverage exact: every core node still maps to exactly one tagged decl; never drop or duplicate a node.

You do NOT re-run F1.5. Ordinary reuse/type-fit plan drift is reviewed by F3 phase A, but the deterministic delivery audit is a hard gate: an `undelivered` node must have neither an `@node` tag nor any Lean declaration matching its planned `lean_name`, and no delivered result may depend on it. Keep plan.json faithful to what you emitted.

=== MISSING-ARCHITECTURE CHECK (escalate before drafting) ===

A `define-local` node is normally build-inline: F2 scaffolds it, F3 proves it. Escalate ONLY when realizing a node would require a SUBSTANTIAL new architectural piece beyond a build-inline def/lemma — return `blocked-missing-architecture` (below). Before escalating, run the MANDATORY PARENT-ABSTRACTION CHECK:

1. **Is there a parent abstraction that fits?** A specialized Causalean primitive failing to fit is NOT grounds to escalate if the parent (`POSystem`/`POManski`/`SWIGGraph`/`SCM.Do`) carries the statement — scaffold against the parent + theorem-local hypotheses/`let`s. Do NOT escalate.
2. **Can the "missing structure" be a proof-internal construction?** A subsystem built by conditioning, finite indexing, or assembling existing primitives is local construction (theorem-local hypotheses + local `def`s), NOT architecture.
3. **Is it a single Mathlib-shaped helper?** Then scaffold a sorry'd lemma in `CausalSmith/CausalSmith/Mathlib/<topic>.lean` — not an escalation.

What ACTUALLY counts as substantial (escalate only after all three fail): a new `class`/`structure` (forbidden in causalsmith mode regardless); a new core `def` with its own ≥3-lemma API reused across ≥2 theorems; a whole module of cluster-scale primitives; or a `reuse` decl the plan names that does not exist AND no parent abstraction carries the statement. `why_substantial` MUST cite the result of each of the three checks.

=== TAUTOLOGICAL PREDICATE WATCH (Lean-encoding faithfulness — still load-bearing) ===

Do NOT make a node vacuous in Lean:
- defining a function with the desired return values and asserting self-equality (`def f := rhs` then `H : f = rhs`);
- a T-block conclusion `P ↔ body(P)` where `body(P)` is the `def`-body of `P`;
- a hypothesis stated as the literal body of an existing `def`;
- packaging the conclusion as a re-arrangement of the hypotheses (so `assumption`/`tauto` closes it);
- OVER-GENERALIZATION: replacing a constructed core object (a class structure, a construction the statement is stated over) with a free variable + a defining/existence hypothesis, so the theorem quantifies over an ARBITRARY object — build the object, do not abstract it;
- ASSUMING THE CONCLUSION: giving an L/T-block a hypothesis that IS its conclusion (`lemma foo (h : P) : P`). Derive the conclusion from genuine upstream inputs (the core's assumptions / an earlier statement's conclusion), never restate the goal as a premise;
- DERIVABLE INTERMEDIATE AS PREMISE: a quantity this block is responsible for deriving (an intermediate on the path to its own conclusion, reached by applying a named lemma to the inputs) is NOT an input — derive it, leave the work to `sorry`. Only a STRICTLY-upstream statement's conclusion (earlier in the core `depends_on` order) may be a hypothesis. This INCLUDES a structural PROPERTY of an object the block constructs or receives (a continuity/convergence/limit, a boundedness, a moment or algebraic identity) that the note DERIVES from that object's construction: derive it in-proof (`have h : … := by sorry`), NEVER thread it as a premise — even if you gave the property its own `def`/`@node`. Having a `def` for a property does NOT make it a hypothesis, and a property the note PROVES is never a free "bookkeeping regularity" premise;
- UNIVERSAL-CONSTANT QUANTIFIER ORDER: when the note states a constant is UNIVERSAL (independent of the model/class parameters — e.g. "a universal numerical constant `c₀`", "an absolute constant"), quantify it OUTERMOST (`∃ c₀, 0 < c₀ ∧ ∀ (model/class params) …`), never after those parameters (`∀ params, ∃ c₀ …`), which silently WEAKENS the claim to a parameter-dependent constant. Mirror the note's stated dependency structure exactly;
- DROPPED CONSTRAINT IN HELPERS: when the top-level statement constrains an object by a predicate, every helper lemma mentioning that object must carry the SAME constraint — thread it down;
- WEAKENED EXISTENTIAL / DROPPED CONCLUSION (statement STRENGTH): render the note's quantifier structure and full conclusion at its STATED strength. Never weaken an existential over a LARGE set (a nonempty OPEN / positive-measure / dense subset, a whole admissible family) to a SINGLE witness; never collapse a membership stated as "the class of objects for which THERE EXIST parameters/witnesses realizing the target" into a fixed-parameter predicate that drops those existentials and the realization/law-membership condition; and never OMIT conjuncts the statement asserts (a stated identity, a uniqueness/recovery clause, a side condition). A tractable proxy that is logically WEAKER than the note is drift, not a faithful encoding — encode the note's actual existential/topological structure, or FLAG it (return early with a query) if that structure is genuinely not renderable, rather than silently emitting the weaker form.

Use a SUBSTANTIVE predicate over the substrate's actual primitives (couple a real quantity to its closed form, not `def`-body to its own unfolding). If unsure whether a planned statement is tautological, FLAG it and return early with a query.

=== UNDER-STRENGTH SIGNATURE WATCH (the opposite failure — unprovable, invisible until F3) ===

- NEVER stub a load-bearing hypothesis as `(_h : True)` — the `sorry` body is the placeholder, the hypotheses are not.
- HYPOTHESIS-COMPLETENESS: when a block's proof will invoke a named substrate lemma, read that lemma's actual signature (`lean_local_search` then `lean_hover_info`) and transcribe the hypotheses it REQUIRES — measurability/integrability side-conditions, an independence/cross-fit witness, a positivity bound, or a strictly-upstream statement's conclusion — drawn from the core's named assumptions. Completeness, not invention: include what the core licenses; add no content beyond it.
- BOOKKEEPING PREMISES ARE NOT "ASSUMING THE CONCLUSION", and you must supply them PROACTIVELY (see the BOOKKEEPING PREMISE POLICY block). A regularity side-condition (integrability/measurability/finiteness) a step needs is honest scope — state it on the signature; prefer to DERIVE a FREE one inline (a bounded measurable function on a probability measure is integrable). Heed the WEAK-OVERLAP CARVE-OUT: integrability of an unbounded inverse-weight/score is NOT free — derive it from the tail/overlap assumption.
- MINIMAX / CONVERSE LOWER-BOUND RISK ENCODING (the junk-value trap): a converse over EVERY estimator is FALSE if encoded with a real `⨆`/`ciSup` over an unbounded risk set (junk value 0) or an ℝ `∫` of a non-integrable loss (junk value 0). Encode well-posed: the witness form `∃ Q ∈ class, rate ≤ risk(Q)`, and either an ℝ≥0∞ `∫⁻` of the loss or an `Integrable` finite-risk premise scoped inside the estimator binder.
- DISTRIBUTION-CLASS AUXILIARY PINNING: when you build a model class, every auxiliary that is a FUNCTIONAL of the law MUST be pinned to the law by its defining identity INSIDE the class, NOT a free `∃`/`∀` parameter (a free auxiliary silently widens the class — a safe-for-the-prover, SILENT defect on a converse).
- UNIFORM-CONSTANT MEMBERSHIP ON A CONVERSE: when the core conclusion is uniform-over-class / "constants independent of n", scaffold the witness membership against a FIXED-constant class (constants as parameters quantified outside n), never the bare per-law `∃`-constant predicate.

=== HARD RULES ===

- **COLD MODE:** never write a proof body; use `:= by sorry`. **REVISE MODE / SCAFFOLD REDIRECT:** never replace a working proof with `sorry` when its declaration signature is unchanged. Preserve unchanged proof bodies as active Lean code; adapt affected proofs when practical; author a new proof for nothing else.
- **CODE-ENFORCED PRIOR-PROOF PRESERVATION:** on revise/redirect rewinds, the orchestrator snapshots every real theorem/lemma proof before this call and restores signature-identical proofs after it. Do not delete, rename, or churn unrelated declarations in an attempt to defeat that preservation. A declaration whose signature genuinely changes receives no automatic proof and may remain `sorry` for Stage 3.
- Statements, assumptions, and definitions match the CORE node verbatim. No silent strengthening, weakening, extra assumptions, or vacuous restatements.
- **Encoding fidelity (literal-unfold rule).** Every Lean encoding must unfold definitionally to the core's math. If the encoding needs extra hypotheses to become equivalent, the encoding is wrong — pick the form whose definitional unfold matches. Look up edge-case semantics (`lean_hover_info`, `lean_local_search`) before committing.
- Definitions are NOT `sorry`ed (full body required); only lemmas/theorems carry `:= by sorry`.
- Do NOT introduce new typeclasses or cluster-scale structures (the missing-architecture kind). CARVE-OUT: a class structure / construction the plan marks `define-local` and a statement is stated over IS built concretely — not replaced by a free variable.
- Edit policy: write only under the `Lean target subdirectory` and `CausalSmith/CausalSmith/Mathlib/`, plus plan.json (sync-back). Do NOT touch `Causalean/` directly — escalate genuine substrate gaps via `blocked-missing-architecture`.
